Adding Modules to a

New module development has different failure modes than greenfield projects. The codebase has history, drift, and established patterns that the new module must respect — or explicitly break with a documented reason.

Part 1 — New Project
Clean slate. LLM and agent work from blueprints. Main risk: premature execution before design is complete.
Part 2 — Existing Project
Codebase has drifted. New module inherits the real state, not the intended state. Main risk: building on a broken foundation.

Before You Write a Single Line

The biggest mistakes in module addition happen before development starts. These disciplines prevent you from building on a foundation that can't support the new module.

✗ What Most Developers Do
→"I need Admin Dashboard" → ask agent to start building immediately
→Agent reads drifted codebase → makes assumptions about what exists
→Admin built on top of missing AuthGuard, no hooks, no types
→Silent failures in production. Auth bypassed. Payment logic duplicated. Types missing.
→10+ iteration cycles to untangle what was actually wrong
✓ The Right Approach
→"I need Admin Dashboard" → LLM designs theoretically first
→Agent audits reality against the theoretical design
→Developer reviews gaps — decides fix or accept
→LLM redesigns for reality + gap decisions
→Agent executes with complete, unambiguous package
1
Run the LLM→Agent→LLM flow — always
Every new module, no matter how small. The three-step flow is not optional — it is the discipline. Skipping it means the agent works from assumptions, not facts.
✗
"It's just a small feature, I'll just ask the agent to add it"
✓
Even a single new hook: LLM designs it → Agent checks what hooks exist → LLM confirms approach
2
Resolve blockers before building
If the Agent gap report flags a BLOCKER, that is not a suggestion — it is a prerequisite. AuthGuard missing means Admin route protection is impossible to do correctly.
✗
Build Admin first, add AuthGuard later → Admin ships with no route protection → security incident
✓
Part A executes first (blockers), then Part B (new module). Always in that sequence.
3
You own the gap decisions — not the LLM
The LLM surfaces gaps. The Agent quantifies them. But the developer decides which to fix and which to accept. Never let the LLM silently absorb a gap decision into the design.
✗
LLM says "I'll work around the missing types" → gap accepted silently, no record, no decision trail
✓
Every gap gets an explicit decision: fix or accept. Accepted risks are logged in the execution package.
4
Commit the current state before anything
Before any remediation or new module work begins, commit everything that currently exists — even the drifted, imperfect state. You need a clean rollback point.
✗
Start remediation on uncommitted work → something breaks → no clean state to return to
✓
git commit -m "chore: snapshot before admin module" → then begin Part A
★ Pre-Module Checklist — Complete Before Any Code
Before starting the LLM→Agent→LLM flow 0 / 6
Committed current codebase state — clean rollback point exists must
Have the original Part 1 blueprint JSON — LLM needs this as reference LLM
Can describe the new module clearly in 2-3 sentences with specific features required
Know which existing modules the new module will need to interact with
Have the actual file tree ready for the agent Agent — run tree src/ in terminal

During Development

Once the execution package is ready and Part A remediation is complete, these disciplines keep the new module consistent with the rest of the codebase.

1
Follow the patterns that actually exist
The new module must match how the existing codebase actually works — not how the blueprint said it would work. If auth components are in lowercase folders, Admin goes in a capitalised folder with a documented reason.
✗
LLM generates Admin with barrel index.ts files — none exist elsewhere, now Admin is inconsistent
✓
Tell agent: "Match existing patterns in Dashboard/ folder — no barrel files, PascalCase components"
2
Never reuse logic — extract shared hooks
If the new module needs payment or subscription logic that currently sits inline in another component, extract it to a shared hook first. Two components sharing inline logic is two sources of truth waiting to diverge.
✗
Admin copies subscription logic from Checkout.tsx → payment logic now exists in two places
✓
Part A extracts useSubscription.ts → Admin and Checkout both import from single source
3
Part A complete and verified before Part B
Remediation is not a background task. Every blocker resolved in Part A is a foundation stone for Part B. Do not start Part B while Part A is incomplete — even partially.
✗
"AuthGuard is 80% done, I'll finish it alongside Admin" → Admin is built against incomplete auth
✓
AuthGuard built → tested → committed → lint passes → then start Admin
4
Scope agent tasks to one module boundary
The agent must stay within the Admin module boundary. If it starts modifying Dashboard or Payment components, stop it. Cross-module changes without an LLM design decision are always wrong.
✗
Agent "helpfully" refactors Dashboard while building Admin → untested changes in a working module
✓
Prompt includes: "Only create files under src/components/Admin/. Do not modify any existing files."
5
Lint after every agent session
Run lint immediately after each agent session — not at the end of the module build. Agents introduce lint errors silently. Catching them per session means you know exactly which session caused it.
✗
5 agent sessions → run lint once at end → 23 errors → don't know which session caused what
✓
Session 1 → npm run lint → 0 errors → commit → Session 2 → lint → repeat
6
When agent and LLM disagree — LLM wins
The LLM designed the module with the full picture — blueprint, gap report, your decisions. If the agent suggests a different approach mid-execution, that is the agent working from local context only. Escalate to LLM for any structural decision.
✗
Agent says "I'll put admin types in the component folder" → accept without checking LLM design
✓
Ask LLM: "Agent suggested X — does this conflict with the final design?" LLM arbitrates.
★ Per-Session Checklist — Run Each Agent Session
Each agent session during module build 0 / 7
Scoped prompt includes: exact files to create, module boundary, "do not modify existing files" Agent
Part A remediation is fully complete and committed before any Part B work must
Run npm run lint after session — 0 errors before next session
Read every file the agent created — no unreviewed output risk
Committed after each verified session — clean bisect history
Agent stayed within module boundary — no cross-module modifications
Any structural agent suggestion checked against LLM final design before accepting LLM

Anti-Patterns

These are the specific mistakes developers make when adding modules to existing projects with AI assistance. Each one compounds the existing drift rather than resolving it.

💀 Building on Unresolved Blockers
What happens Developer skips Part A, asks agent to build Admin Dashboard directly. AuthGuard doesn't exist. Agent implements its own ad-hoc route check inline in AdminLayout.tsx.
Why it's fatal Now route protection logic exists in two places — AdminLayout and wherever AuthGuard eventually gets built. They will diverge. Admin ships with weaker auth than the rest of the app. ✓ Fix: Part A fully resolved and committed before Part B starts. No exceptions.
🔁 Duplicating Logic Instead of Extracting
What happens Admin needs subscription status. Agent copies the inline subscription check from Checkout.tsx into AdminSubscriptionPanel.tsx. Two copies, same logic.
Why it's fatal Next time subscription logic changes — a new plan type, a Razorpay webhook update — the developer must remember to update both places. They won't. Silent inconsistency ships. ✓ Fix: Extract to useSubscription.ts in Part A. Admin and Checkout both import from one source.
🌊 Letting the Agent Drift the Blueprint Further
What happens Agent creates Admin files in src/admin/ instead of src/components/Admin/. Developer doesn't notice. Now there are three naming conventions in the codebase.
Why it's fatal Every future module has more conventions to choose from. RAG context for future agents gets noisier. New developers have no clear pattern to follow. ✓ Fix: Execution package specifies exact paths. Agent prompt says "use only these paths, no variations."
🎲 Accepting Silent Gap Decisions from the LLM
What happens LLM notices the types/ folder is missing and silently designs Admin types inline in each component instead of flagging it as a gap. Developer doesn't realise a decision was made for them.
Why it's fatal Inline types can't be shared. Next module needs User type — it either duplicates it or has no types. The types/ folder gap has now propagated across two modules. ✓ Fix: Every gap must surface in Step 2 and get an explicit developer decision in Step 3.
🏃 Stacking Multiple Module Additions Without Gaps Review
What happens After Admin Dashboard, developer immediately asks agent to also add Analytics Module and Notification Module in the same session.
Why it's fatal Each new module should run the LLM→Agent→LLM flow independently. Stacking them means gaps in Admin affect Analytics which affects Notifications. By the end nothing is cleanly isolated. ✓ Fix: One module at a time. Complete flow, commit, verify. Then start the next module's flow.

→ Risk: premature execution

→ Agent audits reality vs theory
→ Risk: building on broken foundation
→ Main discipline: reconcile before build
→ Codebase = reality (not intended state)
